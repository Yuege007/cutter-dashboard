# 看板配置后端保存与加载方案

## 1. 背景

当前看板作为应用内跳转网页使用。用户从主应用进入看板页面时，看板会获取当前应用账号信息，然后展示对应账号的看板。

现在布局主要保存在浏览器本地缓存中，不同设备之间无法共享。目标是把看板配置保存到后端，让同一账号在不同设备登录后，可以自动恢复或手动加载原配置。

## 2. 目标

实现一套“看板配置快照”能力：

- 点击保存时，把当前看板结构保存到后端。
- 进入页面时，自动拉取当前账号的后端配置并恢复。
- 用户点击加载时，也可以主动从后端重新拉取配置。
- 后端只保存配置，不保存业务接口数据。
- 本地缓存保留为兜底，避免后端不可用时看板完全丢失。

## 3. 保存范围

### 需要保存

看板级配置：

- 看板标识 `dashboardKey`。
- 配置版本 `version`。
- 看板标题 `title`。
- 当前主题 `theme`。

卡片级配置：

- 有哪些卡片。
- 卡片类型 `cardId`。
- 卡片位置 `position.x / position.y`。
- 卡片大小 `size.w / size.h`。
- 卡片锁定显示模式 `lockedMode`。
- 卡片自身配置 `props`。

底部消息配置：

- 手动消息。
- 自动业务消息开关。
- 播放 / 暂停状态是否保存待定。

### 不需要保存

- 卡片接口返回数据。
- loading / error 状态。
- 图表临时数据。
- 自动轮播当前页码。
- 卡片内部临时 hover / active 状态。
- `lastUpdated`。
- 后端业务接口缓存。

## 4. 建议数据结构

```ts
export interface DashboardConfigPayload {
  version: string
  dashboardKey: string
  title: string
  cards: DashboardCardConfig[]
  settings: DashboardSettings
  updatedAt: string
}

export interface DashboardCardConfig {
  cardId: string
  position: {
    x: number
    y: number
  }
  size: {
    w: number
    h: number
  }
  lockedMode?: 'mini' | 'compact' | 'full'
  locked?: boolean
  props?: Record<string, any>
}

export interface DashboardSettings {
  theme: 'light' | 'dark' | 'tech'
  bottomBanner: {
    manualMessages: Array<{
      id: string
      content: string
      createdAt: string
      type?: string
    }>
    autoMessageSettings: Record<string, boolean>
    isPaused?: boolean
  }
}
```

示例：

```json
{
  "version": "1.0",
  "dashboardKey": "tool-cabinet-dashboard",
  "title": "数字看板系统",
  "cards": [
    {
      "cardId": "inventory-alarm-card",
      "position": { "x": 0, "y": 0 },
      "size": { "w": 6, "h": 5 },
      "lockedMode": "full",
      "locked": false,
      "props": {}
    }
  ],
  "settings": {
    "theme": "light",
    "bottomBanner": {
      "manualMessages": [],
      "autoMessageSettings": {
        "inventorySummary": true,
        "zeroStockMaterials": true,
        "channelAlarmSummary": true
      },
      "isPaused": false
    }
  },
  "updatedAt": "2026-04-28T10:00:00.000Z"
}
```

## 5. 后端接口建议

### 获取配置

```http
GET /dashboard/config?dashboardKey=tool-cabinet-dashboard
```

建议后端从登录 token 中识别当前用户，不依赖前端传 `userId` 作为可信身份。

返回：

```json
{
  "code": "OK",
  "message": "查询成功",
  "data": {
    "dashboardKey": "tool-cabinet-dashboard",
    "config": {}
  }
}
```

### 保存配置

```http
POST /dashboard/config
```

入参：

```json
{
  "dashboardKey": "tool-cabinet-dashboard",
  "config": {}
}
```

后端保存维度：

```text
userId + dashboardKey
```

如果后续要支持多个看板，可以扩展为：

```text
userId + dashboardKey + workspaceId
```

### 6.2 `dashboardConfigApi.ts`

职责：

- 封装保存接口。
- 封装加载接口。
- 统一处理接口路径和响应结构。

建议方法：

```ts
export const dashboardConfigApi = {
  getConfig: (dashboardKey: string) => {},
  saveConfig: (dashboardKey: string, config: DashboardConfigPayload) => {}
}
```

### 6.3 `dashboardConfigSerializer.ts`

职责：

- 从当前页面状态生成配置。
- 从后端配置恢复页面。

建议方法：

```ts
export const buildDashboardConfig = () => {}
export const restoreDashboardConfig = (config: DashboardConfigPayload) => {}
```

生成配置时读取：

- `cardStore.activeCardsList`
- `dashboardTitle`
- `themeStore.currentTheme`
- `bottom-banner-data`
- `bottom-banner-auto-message-settings`

恢复配置时处理：

- 清空当前卡片。
- 按 `cardId` 添加卡片。
- 恢复卡片位置。
- 恢复卡片大小。
- 恢复 `lockedMode`。
- 恢复 `props`。
- 恢复标题。
- 恢复主题。
- 恢复底部消息配置。

## 7. Dashboard 接入点

需要修改：

```text
vue-dashboard/src/views/Dashboard.vue
```

### 7.1 进入页面自动加载

当前逻辑是恢复用户本地布局。后续改为：

```text
恢复账号信息
↓
请求后端看板配置
↓
有后端配置：恢复后端配置
↓
没有后端配置：恢复本地配置
↓
都没有：使用默认空看板
```

### 7.2 点击保存

当前 `handleSaveLayout` 保存本地布局。后续改为：

```text
buildDashboardConfig()
↓
dashboardConfigApi.saveConfig()
↓
保存成功后同步本地缓存
↓
提示“配置已保存”
```

### 7.3 点击加载

当前 `handleLoadLayout` 从本地恢复。后续改为：

```text
dashboardConfigApi.getConfig()
↓
有配置：restoreDashboardConfig()
↓
没有配置：提示“暂无后端配置”
↓
接口失败：尝试本地缓存兜底
```

## 8. 本地缓存策略

后端配置作为主数据源，本地缓存作为兜底。

保存成功后：

- 保存后端。
- 同步保存一份本地布局。

加载时：

1. 优先后端配置。
2. 后端无配置或失败时，尝试本地配置。
3. 本地也没有时，显示空看板或默认看板。

## 9. 兼容现有逻辑

当前已有：

- `layoutPersistence.saveUserLayout`
- `layoutPersistence.restoreUserLayout`
- `cardStore.addCard`
- `cardStore.updateCardSize`
- `cardStore.clearAllCards`

新方案可以先复用这些能力，不需要一次性删除旧逻辑。

建议第一阶段：

- 新增后端配置保存 / 加载。
- 保留本地布局保存。
- 让保存按钮同时保存后端和本地。
- 让加载和进入页面优先读后端。

第二阶段再评估是否减少本地布局逻辑。

## 11. 开发顺序建议

1. 前后端确认 JSON 结构。
2. 前端新增类型文件。
3. 前端新增配置序列化 / 反序列化方法。
4. 前端新增 API 封装。
5. 修改保存按钮，先接保存接口。
6. 修改加载按钮，接加载接口。
7. 修改页面初始化，自动加载后端配置。
8. 保留本地缓存兜底。
9. 构建验证。
10. 多设备登录验证。
